**Date:** 2026-05-11
**Status:** Phase 1 complete (diagnosis only). Awaiting Travis greenlight before Phase 2 code.

## Why this is a real code bug (option (a), not (b))

Each surface has its OWN copy of the threshold logic. There is no central function and no shared constant. Even if the band were updated in the spec, six different files would still need code edits to enforce it consistently. Spec-only path is insufficient.

**Why Overview AI Briefing said "sustainable" despite the prompts saying "below healthy":**
The Briefing is the SYNTHESIZED output of all 7 agents (per `api/advisory/route.ts` orchestration). The synthesis step produces a final summary string that smooths over individual-agent disagreements. The Staffing Advisor agent on its own (visible on Advisory Panel tab) DID say "below the healthy range" — matches its hardcoded prompt. So the briefing isn't an exception to the rule; it's a synthesis artifact that happens to land closer to reality.

### Notable existing year-aware logic that's pointing the wrong direction

`api/advisory/route.ts:144-145` already has Y1-aware concerns but interpreted backward:

```
RED FLAGS (trigger amber or red):
- Personnel % >80% in any year
- Personnel % <70% in Y1 (under-staffed for a startup)
```

The prompt treats <70% in Y1 as a red flag (under-staffed). Per Travis's C-decision, 65–72% in founding year is normal — not a red flag. The Staffing Advisor prompt actively codifies the wrong intuition.

## 1.2 Proposed fix shape — centralized helper

Create `src/lib/healthThresholds.ts` exporting a single function consumed by all six surfaces:

```ts
export type PersonnelHealthVerdict = 'meets' | 'approaching' | 'fails'

export interface PersonnelHealthEvaluation {
  verdict: PersonnelHealthVerdict
  band: { lower: number; upper: number; label: string }   // e.g., "Founding year 65–72%"
  explanation: string
}

export function evaluatePersonnelPctHealth(
  pct: number,
  year: number,
  schoolStatus: 'planning' | 'authorized' | 'exported' = 'planning',
): PersonnelHealthEvaluation
```

**Consumers (after refactor):**
- Overview tile: read `verdict` to set `data-status`, read `band.label` for tooltip
- Staffing Advisor prompt: inject `band.lower`-`band.upper` into the system prompt at render time, not hardcoded
- Ask SchoolLaunch prompt: same injection
- Staffing tab UI copy: read `explanation` for the inline warning text
- PDF export: read `verdict` + `band` for the rendered range text
- Dev showcase: optionally consume the same helper (or leave as outlier)

**Band proposal (Travis's C-decision):**
- Year 1: 65–72% healthy (founding-year)
- Year 2: 68–75% healthy (ramp-up)
- Year 3+: 72–78% healthy (steady state)

Open question: should Y2 be its own band, or just blend founding → steady-state? Recommend two-band model (Y1 founding, Y2+ steady-state) for simplicity. Surface to Travis.

## 1.3 Decisions to surface to Travis

### D1 — Band granularity
- (A) Two bands: Y1 founding (65–72%), Y2+ steady-state (72–78%)
- (B) Three bands: Y1 (65–72%), Y2 (68–75%), Y3+ (72–78%)
- Recommendation: (A) — simpler, easier to defend to Commission. Y2 ramp is largely captured by enrollment growth pushing personnel % up naturally.

### D2 — Bottom of founding band
- 65% is the spec's stated lower bound. But the audit's Evergreen is at 65.3% — barely inside. Should the lower bound be 62% or 60% to give meaningful slack? Or hold at 65%?
- Recommendation: hold at 65%. Below 65% genuinely is under-staffed for a startup.

### D3 — "Approaching" semantics
- Current Overview tile has `<=80 = approaching` (warning). Should approaching apply on the LOW side too (e.g., <67% in founding year = "approaching low")?
- Recommendation: yes, symmetric approaching bands on both sides. Otherwise tile color logic stays one-sided and inconsistent with the AI prompts' "<70% in Y1 flag" framing.

### D4 — School status awareness
- Should `schoolStatus` (planning/authorized/exported) affect the verdict? E.g., authorized schools might warrant tighter bands.
- Recommendation: NOT for v1 of this helper. Add later if needed.

### D5 — Helper location
- `src/lib/healthThresholds.ts` (new file) — recommended
- `src/lib/calculations.ts` — alternative, but file is large already
- `src/lib/budgetEngine.ts` — would conflate engine math with UX threshold

Recommendation: new file `src/lib/healthThresholds.ts`. Single responsibility.

### D6 — Tooltip language consistency
- Whichever band/explanation gets adopted should land verbatim in:
  - Overview tile tooltip
  - Staffing tab inline warning
  - The "What does this mean?" copy in the explainer panel (if any)
  - Any future onboarding context-help

Decision: same string everywhere, sourced from the helper.

### D7 — Dev showcase outlier
- `Showcase.tsx:95` says "75–80%" — different from production. Leave alone? Update?
- Recommendation: out of scope. Dev component, separate from product. Flag in a separate doc-hygiene pass.

### D8 — F3-style coherence check for AI prompts
- After Phase 2 lands, regenerate Evergreen's advisory_cache and check that no agent prose includes obsolete "72-78%" string literals not derived from the helper. Same pattern as the R-ENR-01 F3 prose fix.

## 1.6 Open scope question

The 4-surface inconsistency surfaced via Personnel %, but the same hardcoded-threshold-distributed-across-surfaces pattern probably exists for other health metrics:

- Days of Cash thresholds (Stage 1 ≥30, Stage 2 ≥60) — partially centralized in `computeFPFScorecard` but the Overview tile separately calls `reserveStatus(daysOfCash)` (per `dashboard/page.tsx:197`)
- Total Margin thresholds (S1 ≥-5%, S2 ≥0%) — likely similar pattern
- Current Ratio, Enrollment Variance, etc.

**Decision needed:** Is F-001/F-011 a one-metric fix (just Personnel %), or do we extend the helper pattern to cover all health metrics in the same pass?

- (A) One-metric scope: ship `evaluatePersonnelPctHealth` only. Smaller, faster, less risk.
- (B) All-metric scope: ship `healthThresholds.ts` with `evaluatePersonnelPctHealth`, `evaluateDaysOfCashHealth`, `evaluateTotalMarginHealth`, etc. Larger refactor.

Recommendation: (A) for May 19. Extend to (B) in a follow-up after the threshold pattern is proven on Personnel %.
